# 運用設計

## 質問ログ分析

質問ログ分析は、漁協職員や自治体職員がどの制度分野で困っているかを把握し、FAQ改善、研修テーマ抽出、支援施策改善、情報源追加に活用するための機能です。

分析項目は、期間別質問数、カテゴリ別質問数、所属別質問数、ロール別質問数、頻出キーワード、不足資料、回答信頼度分布、回答評価、未解決質問、FAQ候補、研修テーマ候補です。

## 遊漁船業の運用

カテゴリ12は安全管理に関わるため、遊漁船業法、施行令、施行規則、都道府県登録手続、業務主任者講習資料、業務規程例、安全管理通知、事故報告関係資料、小型船舶操縦士資料、特定操縦免許資料を更新管理対象にします。

## Phase 2 以降

案件管理では、遊漁船業、遊漁船業者登録、遊漁船業務主任者、遊漁船安全管理、体験漁業、観光漁業、海洋レジャー、漁業者兼業、質問ログ分析、研修テーマ、FAQ改善、情報源追加を相談区分として扱います。

文書生成テンプレートには、遊漁船業者登録相談メモ、主任者確認メモ、安全管理チェックシート、漁協向け兼業指導メモ、漁港出航確認メモ、事故初動確認メモ、質問ログ分析レポート、研修テーマ案、FAQ改善提案書、不足資料追加提案書を追加します。

## Phase 3 資料更新運用

資料ごとに、最終確認日、次回確認日、更新周期、取得元、更新担当者、改正有無、影響するFAQ、影響する回答テンプレート、影響する案件を管理します。更新期限を過ぎた資料は `/admin/documents` で警告し、AI回答の根拠に使う場合も注意喚起します。

資料更新時は旧版を上書きせず、`document_versions` に新版を追加します。可能な場合は旧版テキストと新版テキストの差分を保存し、最低限、旧版と新版を並べて閲覧できる状態にします。

## 監査と障害時確認

監査ログは `/admin/logs` で確認します。ログイン、質問、AI回答、資料検索、資料閲覧、資料登録、資料更新、案件登録、文書生成、権限変更、設定変更、ログ検索を記録対象とします。

障害時は、`/api/health`、Vercelログ、Supabaseログ、OpenAI APIエラー、監査ログ、直近の資料更新履歴を確認します。利用者へ機密情報を含むエラーメッセージを表示しない方針です。

## バックアップと復元

削除は論理削除とし、`deleted_at`、`deleted_by`、`delete_reason` を保存します。資料は旧版を `document_versions` に保持し、案件は対応履歴と添付資料を残します。

復元手順:

1. `/admin/logs` で削除操作、実行者、対象ID、理由を確認する。
2. 管理者以上が復元対象を選択する。
3. `deleted_at` を解除し、`restored_at`、`restored_by` を記録する。
4. `backup_restore_events` と `audit_logs` に復元結果を保存する。
5. 復元後にRLS、公開範囲、最新版バージョン、検索チャンクを確認する。

バックアップ手順:

1. Supabaseの定期バックアップ設定を確認する。
2. 重要更新前にDBバックアップとStorage対象bucketのエクスポートを取得する。
3. 復元テスト結果を `backup_restore_events` に記録する。
4. 個人情報を含むバックアップの保管先、暗号化、アクセス権を確認する。

## GitHub容量管理と外部保存

GitHubにはコード、SQLマイグレーション、最小seed、設計文書、テストだけを置きます。法令PDF、写真、音声、証憑、質問ログ、AI回答履歴、Embedding、DBダンプ、バックアップはGit管理せず、Supabase又は外部Object Storageへ保存します。

運用時は以下を確認します。

1. `npm run check:repo-size` が成功する。
2. `.env`、PDF、DBダンプ、バックアップ、実案件データがGit差分に含まれていない。
3. `/admin/storage` でbucket用途とアクセス方針を確認する。
4. `/admin/backups` でバックアップ手順を確認する。
5. `/admin/archives` でアーカイブ候補と転送状態を確認する。

Storage転送で失敗した場合は、`storage_transfer_jobs` の `status`、`error_message`、対象bucket/path、監査ログを確認します。
