# セキュリティ設計

## ロール

- 一般利用者: 公開資料検索、質問、自分の質問履歴、お気に入り
- 漁協職員: 自分の相談記録、自所属の共有案件閲覧
- 漁協管理者: 自所属職員の質問ログ集計、FAQ改善候補、案件管理
- 自治体職員: 漁協指導案件、内部メモ、自治体資料閲覧
- 自治体管理者: 全体ログ集計、所属別・カテゴリ別分析、研修テーマ抽出、不足資料集計
- 管理者: 資料、カテゴリ、ユーザー、プロンプト、更新管理
- システム管理者: RLS、ログ、システム設定を含む全権限

## RLS 方針

公開資料は全利用者、漁協内部資料は所属漁協職員以上、自治体内部資料は自治体職員以上、管理資料は管理者以上に制限します。質問履歴は本人が閲覧でき、所属管理者や自治体管理者は同意と利用目的の範囲内で集計分析できます。

SQL Editor向けの `supabase/manual-setup/05_policies.sql` は、主要テーブルのRLSを有効化し、すべての `create policy` の直前に `drop policy if exists` を置きます。これにより、手動実行中に `policy already exists` で止まりにくくしています。

フロントエンドだけでなくDB側のRLSで以下を強制します。

- 公開資料: 全利用者
- 漁協内部資料: 漁協職員、自治体職員、管理者、システム管理者
- 自治体内部資料: 自治体職員、管理者、システム管理者
- 管理資料: 管理者、システム管理者
- 非公開案件: 案件担当者、関係者、管理者、システム管理者
- `qa_sessions`: 通常の質問履歴は本人のみselect可能
- `qa_messages`: 本人の質問セッションに属するメッセージのみselect可能
- `qa_sources`: 本人の質問セッションに属する根拠のみselect可能
- 所属管理者: 個別本文ではなく、匿名化・マスキング済みの集計関数のみ閲覧可能
- 個別質問ログ: 管理者以上が閲覧理由を入力し、`record_question_log_access` により `audit_logs` と `question_log_access_events` へ記録した場合のみ閲覧する
- `favorites`: 本人のみ
- `consultation_cases`: 案件担当者、関係者、管理者、システム管理者
- `documents` / `document_chunks`: 資料公開範囲と利用者ロールに従う
- `audit_logs`: 管理者、システム管理者のみ

論理削除済みの資料・案件は通常検索結果から除外し、復元操作は管理者以上に限定します。

## 監査ログ

質問ログ閲覧、質問ログ分析、個別ログ閲覧、マスキング解除閲覧、FAQ候補作成、研修テーマ作成、不足資料抽出は監査ログ又は分析イベントに保存します。

Phase 3で記録する操作は、ログイン、ログアウト、質問、AI回答、資料検索、資料閲覧、資料登録、資料更新、資料削除、案件登録、案件更新、文書生成、文書出力、ユーザー追加、権限変更、設定変更、ログ検索です。期間、ユーザー、操作種別、資料、案件、IPアドレス、結果で検索できる設計です。

## 個人情報保護

AIへ送信する前に、氏名、電話番号、メールアドレス、住所、口座情報、個別事業者名、非公開案件名、船名、登録番号を検知し、マスキング前後を利用者に表示します。利用者が確認した後、マスキング後の内容のみをAIへ送信します。

元データは権限を持つ利用者のみ閲覧可能にし、AIへ送った内容は監査ログへマスキング済み要約又はハッシュとして保存します。機密情報をログへ平文保存しません。

質問文は `raw_text`、`masked_text`、`ai_sent_text` を分けて保存します。管理者分析画面は `masked_text` 又は集計値を使用し、原則として `raw_text` を表示しません。

## AI回答の安全策

- 根拠資料がない回答は警告する。
- 旧版資料のみ、更新期限切れ資料、内部資料のみを根拠にする場合は警告する。
- 個別許可、行政財産使用、補助金交付、漁業権行使、採捕規制は所管確認が必要と表示する。
- 根拠資料リンクと引用箇所を必須表示する。
- 引用がない文章はカード上で識別可能にする。
- 利用者評価「役に立った」「要修正」と修正理由を保存する。

## Phase 4 追加方針

- OCR: 外部OCRへ送信する前に、対象資料の公開範囲、個人情報、契約条件、保存先を確認する。
- 音声入力: ブラウザ標準機能は端末側処理に依存する。外部API利用時はサーバー側から送信し、APIキーをブラウザへ露出しない。
- 写真添付: EXIF、位置情報、人物、車両番号、船名、口座情報の写り込みを確認する。機密写真は公開bucketへ保存しない。
- FAQ候補: 利用実績から自動生成しても、自動公開は禁止する。
- オフライン: 機密資料のオフライン保存は既定で無効にできる設計とし、共有端末では保存しない。
- 外部連携: OAuth権限、API利用量、送信データ、監査ログ、撤回手順を確認してから有効化する。

## GitHubと外部Storage

GitHubには実資料、添付、質問ログ、AI回答履歴、Embedding、DBダンプ、バックアップ、個人情報を保存しません。これらはSupabase PostgreSQL、Supabase Storage、又は外部Object Storageへ保存し、RLS、署名付きURL、監査ログ、保存期限で管理します。

資料登録、写真添付、アーカイブ、バックアップは `lib/storage` のStorageAdapter経由で実装します。外部Storageへ切り替える場合も、APIキーや認証情報をクライアントへ露出せず、サーバー側環境変数で扱います。

`.env`、サービスロールキー、OpenAI APIキー、外部StorageキーはGit管理禁止です。`npm run check:repo-size` で混入を検知します。

## Supabase Storage

`documents`、`attachments`、`archives`、`backups`、`consultation-attachments`、`generated` bucketはprivateを基本とします。PDFや添付画像は公開URLではなく、権限確認後に短時間の署名付きURLで表示します。

## service role key

`SUPABASE_SERVICE_ROLE_KEY` は `lib/supabase-admin.ts` を通じてサーバー側API Route又はServer Actionでのみ使用します。`NEXT_PUBLIC_` を付けず、クライアントコンポーネント、ブラウザ、ログ、GitHubへ露出させません。

## Supabase Auth

Supabaseの管理アカウント及びProjectは運営者が手動で作成します。アプリはSupabase Project自体を自動作成しません。

アプリ利用者はSupabase Authで管理し、自由サインアップではなく管理者招待制を基本とします。管理者はメールアドレス、表示名、所属 `organization_id`、ロールを指定して招待し、`users`、`user_organizations`、`user_roles` に反映します。

ログイン方式はメール・パスワード又はマジックリンクを許可します。初回ログイン時又は規約改定時は利用規約、プライバシーポリシー、質問ログ分析への同意を求め、`user_consents` に保存します。同意しない利用者は質問機能を利用できません。

セッションは `@supabase/ssr` のCookieで保持し、Middlewareで期限更新します。保護画面は未ログイン時に `/login`、現行版への未同意時に `/consent` へ誘導します。`/admin` は `admin`、`system_admin`、`super_admin` のみに限定し、資料登録、ユーザー招待、個別質問ログ閲覧APIはCookieから取得した本人の所属とDBロールを再確認します。リクエスト本文のロール値や管理用固定トークンは権限判定に使用しません。

Service Workerはログイン後の画面、質問、資料、案件、管理画面をCache Storageへ保存しません。ログアウト後のオフラインキャッシュから機密画面が表示されることを防ぐため、公開のログイン・規約・プライバシー画面と静的アセットだけをキャッシュします。
